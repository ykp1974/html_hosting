LLMによるドキュメント破壊のメカニズム

Microsoft Research 論文「Delegate52」分析報告

GPT/Gemini/Claudeに共通する落とし穴？解説動画
⚠️ 警告：委譲ワークフローの罠

最先端LLMへのタスク委譲（編集→復元）を繰り返すと、ドキュメントは徐々にではなく「突発的（崖効果）」に破壊されます。特に情報の「削除」ではなく、気づきにくい「意味の改ざん」が深刻なリスクとなります。

1. モデル別：復元精度のパフォーマンス

20回の編集サイクル後、フロンティアモデルでも平均25%の内容が破壊される。

順位 モデル名 評価
1位 Gemini 1.5 Pro 最も安定しており、復元精度が最も高い。
2位 Claude 3.5 Sonnet 高性能だが、Geminiには一歩及ばず。
3位 GPT-4o 初期段階から微細な破壊が発生しやすい。

2. 領域別の耐性（得意・苦手）

✔ 高耐性（得意）
  • Pythonコード

※20回後も98%維持

✖ 低耐性（苦手）
  • JSON / DBスキーマ
  • スプレッドシート
  • 日常文書・レシピ

3. 精度劣化を加速させる4要因

  • ドキュメント長： 1万トークン超で改ざんリスク急増。
  • ノイズ： 無関係なファイルが多いと精度低下。
  • ツールの逆効果： 修正ツール導入で逆にミスが増える。
  • マルチターン： 対話が重なるほど劣化が激化。

4. 失敗のパターン：崖効果と改ざん

📉 崖効果（クリティカル・エラー）

精度は徐々に落ちるのではなく、特定のステップで突然10ポイント以上急落する（全失敗の80%以上）。

✍ 不可視の改ざん

フロンティアモデルは「削除」せず「もっともらしく書き換える」ため、人間が目視でミスを発見するのは困難。

実践的な対策

完全な自律委譲は不可能。Git等を用いた「差分確認（Diff Check）」を必須化し、
「賢いモデルほどもっともらしい嘘を吐く」ことを前提とした監視体制を構築せよ。

※将来のGPT-5等でもこの構造的限界には注意が必要。
